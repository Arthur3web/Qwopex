# Qwopex → Суперапп. Дорожная карта реализации

Документ описывает архитектуру каркаса (уже внедрён) и детальный план
дальнейшей реализации. Стек согласован: **Vanilla JS + дизайн-токены**,
zero-build, строгий CSP, framework-agnostic мини-аппы. Без React/MUI.

---

## 0. Что уже сделано (каркас)

```
index.html        — шелл: общий SVG-спрайт + шапка + точка монтирования #view
manifest.json     — id, scope, shortcuts, maskable SVG-иконка, categories
icon.svg          — maskable-иконка (safe-zone 80%)
sw.js             — network-first навигация + SWR статика + network-only API
styles.css        — дизайн-токены (:root / body.light) + стили шелла/лаунчера
js/
  app.js          — шелл: тема, лаунчер, хэш-роутинг, лайфцикл мини-аппов,
                    тосты, регистрация и обновление SW
  sdk.js          — общий слой: утилиты, санитайзер, storage, шина, auth
  registry.js     — реестр мини-аппов (ленивый import)
  apps/
    ads.js        — «Объявления» (порт script.js под контракт мини-аппа)
    wallet.js     — «Кошелёк» (заглушка)
    chats.js      — «Чаты» (заглушка)
    market.js     — «Маркет» (заглушка)
```

**Контракт мини-аппа** (`default export`):

```js
export default {
  mount(root, ctx) {},   // отрисовать UI в root
  unmount() {},          // снять слушатели/таймеры, очистить root
  onRoute(subpath) {},   // (опц.) реакция на под-маршрут без ремоунта
};
```

`ctx` = `{ user, subpath, bus, navigate(hash), toast(msg), scrollTop() }`.

**Маршрутизация:** хэш-based — `#/` лаунчер, `#/ads`, `#/ads/create`,
`#/wallet`, `#/chats`, `#/market`. Выбрано вместо History API, т.к. не
требует серверных rewrite-правил для статики.

## Этап 1. Стабилизация каркаса (1–2 дня)

- [ ] **Проверить в браузере**: установка PWA, офлайн-режим, обновление SW
      (плашка «Обновить» → reload), навигация между мини-аппами, тема.
      Запуск: `python3 -m http.server 8080` → http://localhost:8080
      (SW требует http/https, не file://).
- [ ] **Растровые иконки**: добавить `images/icon-192.png` и `icon-512.png`
      (часть платформ предпочитает PNG для сплэша/ярлыков). Прописать в
      manifest рядом с `icon.svg`. SW прекэширует устойчиво (allSettled).
- [ ] **Lighthouse PWA-аудит**: добиться «installable» без ворнингов.
- [ ] **Кнопка «назад» ОС/жест**: убедиться, что `hashchange` корректно
      возвращает на `#/` (history уже работает через якоря).
- [ ] **Доступность**: фокус-менеджмент при смене мини-аппа (перевод фокуса
      на заголовок), `aria-live` для тостов (уже `role=status`).

- [ ] **Заменить `alert/confirm`** в ads.js на `<qx-modal>` (сейчас уже
      переведено на `ctx.toast` для сообщений; удаление ещё на `confirm`).
- [ ] Документировать токены и компоненты в этом репозитории.

## Этап 3. Авторизация (Telegram) + бэкенд (3–5 дней)

- [ ] **Бэкенд для проверки подписи Telegram Login** (Node/Go): эндпоинт
      `POST /api/auth/telegram` валидирует `hash` по HMAC с токеном бота,
      выдаёт сессионную httpOnly-cookie или JWT.
- [ ] **Включить виджет**: раскомментировать блок в `sdk.js`, задать
      `TG_BOT_USERNAME`, отрисовать `#telegram-login` на экране логина
      (добавить login-view в шелл, показывать при `auth.getUser() === null`).
- [ ] **CSP**: при появлении API на другом домене расширить `connect-src`
      (сейчас `'self'`); виджет уже разрешён (`script-src https://telegram.org`,
      `frame-src https://oauth.telegram.org`).
- [ ] **SW**: запросы `/api/*` уже network-only — проверить, что cookie/JWT
      проходят (credentials: 'include' на fetch со стороны приложения).
- [ ] Убрать `FAKE_USER` из `sdk.js` (оставить под флагом `DEV`).

## Этап 4. Мини-апп «Объявления» — продакшн (4–6 дней)

- [ ] **Хранилище → API**: заменить `createStorage("ads")` на REST/GraphQL

- [ ] **Загрузка медиа**: реализовать `input[type=file]` (сейчас заглушка) —
      превью, ресайз на клиенте, загрузка в объектное хранилище, прогресс.
- [ ] **Пагинация/бесконечный скролл** списка, серверный поиск и фильтры.
- [ ] **Модерация**: статусы с бэкенда, вебхуки/поллинг обновлений.
- [ ] **Оффлайн-очередь**: создание объявления офлайн → Background Sync
      (`sync` event в SW) с дозагрузкой при появлении сети.

## Этап 5. Платформенные возможности супераппа (5–8 дней)

- [ ] **Push-уведомления**: Web Push (VAPID), `push`/`notificationclick` в SW,
      разрешение запрашивать по действию пользователя, не на старте.
- [ ] **Межмодульное взаимодействие** через `ctx.bus`: напр. «написать
      продавцу» из Объявлений открывает Чаты с предзаполненным диалогом
      (`bus.emit("chats:open", {peerId})` + deep-link `#/chats/<id>`).
- [ ] **Share Target** (manifest): принимать расшаренные изображения/текст
      прямо в форму создания объявления.
- [ ] **Единый платёжный слой**: Кошелёк предоставляет `ctx`-метод
      `pay({amount, purpose})`, которым пользуются другие мини-аппы
      (продвижение объявления, покупка в Маркете).
- [ ] **Разделение кэшей по мини-аппам** в SW (префиксы), чтобы обновление
      одного модуля не инвалидировало остальные.

## Этап 7. Качество и эксплуатация (сквозной)

- [ ] **Тулчейн проверки**: поднять локальный Node LTS (текущий v10 не умеет
      ESM/линт). Подключить ESLint + Prettier (конфиги уже по стилю кода).
- [ ] **CI**: lint + headless-проверка установки PWA (Playwright).
- [ ] **Версионирование SW**: бамп `VERSION` в `sw.js` на каждом релизе

- [ ] **Тесты**: unit на `sanitizeHtml`/`clampInt`, e2e на ключевые сценарии.

## Принятые архитектурные решения

| Решение | Почему |
|---|---|
| Vanilla + токены, без React/MUI | zero-build, строгий CSP без `unsafe-inline`, минимальный вес, framework-agnostic мини-аппы |
| Хэш-роутинг | не нужны серверные rewrite-правила для статического PWA |
| Мини-аппы как ленивые ES-модули | загрузка только при открытии, изоляция, независимый релиз |
| SW: network-first навигация | пользователь всегда получает свежий shell, офлайн — фолбэк из кэша |
| SW: SWR для статики | мгновенный отклик + фоновое обновление ассетов |
| SW: без авто-`skipWaiting` | обновление по согласию (плашка), чтобы не смешать версии модулей |
| `connect-src 'self'` пока | бэкенда нет; расширим точечно при интеграции API |

## Риски

- **Один SW на все мини-аппы**: ошибка в стратегии кэширования затрагивает
  всю платформу — покрыть тестами, версионировать.
- **`document.execCommand` в редакторе** устарел — со временем заменить на
  Selection/Range API или contenteditable-обёртку.
- **localStorage как временное хранилище** объявлений — это только дев-режим,
  до Этапа 4 (миграция на API).
